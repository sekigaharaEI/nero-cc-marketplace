# 风险闸门

在做任何自动化代码修改前，先通过这些闸门。

## 可自动继续

只有同时满足以下条件时，才允许自动继续：

- bug 有清晰复现路径或明确失败证据
- 范围只落在一个仓库和一个主要模块
- 修复动作小且局部
- 修改不涉及认证、支付、权限、数据迁移或跨服务协议
- 至少存在一种确定性的验证方式
- 预期行为没有歧义

## 停止并请求人工确认

命中任意一条就立即停止：

- bug 描述含糊或前后矛盾
- 存在多个都说得通的根因，且会导向不同修复方案
- 修复跨多个模块、多个服务，或影响链路较长
- 需要 schema 迁移、数据修复或运维脚本
- 修改会影响认证、支付、权限、缓存、并发或公开 API 行为
- 验证依赖不稳定的外部系统，或测试覆盖不清楚
- 已确认是 bug，但当前不适合修复

## 修复质量线

进入修改阶段后，至少满足以下要求：

- 优先复用现成失败测试
- 尽量补一个能证明回归被修掉的小测试
- diff 中不要混入无关重构
- 最终状态前必须有干净的定向验证结果

## 配置优先级

- 项目配置优先于 skill 默认值
- 如果项目配置和真实 bug 证据冲突，以真实证据为准
- 如果关键字段缺失，不要猜，先补配置或请求人工确认
