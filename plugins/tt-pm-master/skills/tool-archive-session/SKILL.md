# 会话进度归档

默认把归档理解为“更新项目锚点文件并补齐长期记忆”，不是写泛泛总结。

## 先读

1. `00_Project_Context/alignment.md`
2. `00_Project_Context/project_context.md`
3. `00_Project_Context/task.md`
4. `00_Project_Context/迭代记录.md`
5. `00_Project_Context/next.md`

如果本轮改过样稿，再看最新一周的：

- `终稿`
- `效果验证`
- `投稿复盘`

## 记忆覆盖清单

归档前，先确认本轮能回答这 7 个问题：

1. 当前正在做什么
2. 项目背景是什么
3. 已完成了什么
4. 还没完成什么
5. 达成了哪些观点和共识
6. 当前有哪些稳定偏好
7. 下次开新会话时第一步该做什么

如果有任一项答不出来，就说明锚点文件还没写完整。

## 归档顺序

1. 更新 `task.md`
   - 勾掉已完成项
   - 把新的进行中和下一步写清楚
2. 更新 `迭代记录.md`
   - 写本轮做了什么
   - 写得出了什么结论
   - 写下轮优先动作
3. 更新 `next.md`
   - 只保留下一轮最高优先级
   - 改成可直接执行的动作
4. 只在核心共识变化时更新：
   - `alignment.md`
5. 只在背景或偏好变化时更新：
   - `project_context.md`
6. 只在阶段方案变化时更新：
   - `实施计划.md`

## 归档规则

- 拒绝模糊总结，必须写具体变化。
- 如果本轮没有改变核心目标和边界，不要改 `alignment.md`。
- 如果本轮出现了新的稳定偏好，必须写进 `project_context.md`。
- `实施计划.md` 不是默认必改文件，只在阶段方案真的变化时更新。
- 如果本轮没有推进样稿，不要假装写出内容成果。
- 避免把同一句话重复写进所有文件。
- `task.md` 负责已完成/未完成，`迭代记录.md` 负责本轮变化，`next.md` 负责下一步，`project_context.md` 负责长期背景和偏好。
- 结束时给用户一句下次开场提示。

## 触发判断清单

判断每个文件是否要改，按下面的样例对照，不要凭语感。

### 「核心共识变化」— 必改 `alignment.md`
- 目标用户/受众改变（例：从老人转向年轻自媒体人）
- 商业模式或核心交付物改变
- 当前阶段的范围/MVP 边界改变
- 关键约束改变（例：从开源改为闭源、从竖屏改为横屏）

不算：用词替换、单段大纲调整、单次写作风格选择。

### 「稳定偏好」— 必改 `project_context.md`
- 用户重复表达 2 次以上的审美/工作流偏好
- 明确说「以后都 X」「我不喜欢 X 风格」「永远不要 X」
- 工作流约定（例：先大纲再正文、先看终稿再讨论）

不算：本轮内单次的用词或顺序调整。

### 「阶段方案变化」— 必改 `实施计划.md`
- MVP 功能清单增减
- 阶段拆分方式改变（例：一阶段做完 → phase 1 / phase 2）
- 里程碑时间线大幅调整
- 优先级顺序整体重排

不算：单轮的具体任务推进、子任务拆细。

### `alignment.md` 改动后的级联检查
改了 alignment 必须顺手过一遍：
1. `task.md`：不符合新共识的待办挪到 backlog 或 phase N，不要直接删
2. `next.md`：仍指向旧方向的改成新方向的第一步
3. `迭代记录.md`：必须写明「今日决策：X→Y，原因 Z」，留可追溯证据

## 默认收尾指令

把下面任一句都视为触发：

- `结束本次会话，帮我归档。`
- `按项目归档规则更新锚点文件。`
- `收尾并更新 alignment、task、next、迭代记录。`
