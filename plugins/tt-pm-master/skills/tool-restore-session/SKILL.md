# 恢复项目上下文

默认把恢复理解为“读锚点文件并恢复长期记忆 + 当前状态”，不是复述全部历史。

## 先读

1. `00_Project_Context/alignment.md`
2. `00_Project_Context/project_context.md`
3. `00_Project_Context/迭代记录.md`
4. `00_Project_Context/next.md`
5. `00_Project_Context/task.md`

只有在需要理解阶段方案时，才补读：

- `00_Project_Context/实施计划.md`

如果需要继续具体写作，再看最新一周的：

- `终稿`
- `效果验证`
- `投稿复盘`

## 恢复目标

恢复后必须能回答：

- 当前唯一目标是什么
- 项目背景和长期偏好是什么
- 当前正在做什么
- 已完成了什么、没完成什么
- 当前这轮到底在验证什么
- 已达成了哪些观点和共识
- 现在最优先的下一步是什么
- 是否需要继续最新一周样稿

## 输出格式

恢复完成后，向用户给出 4 点：

1. 当前阶段一句话总结
2. 当前正在做什么 + 已完成/未完成的关键项
3. 当前共识和偏好
4. 建议直接继续的下一步

第 3 点必须满足：
- 至少复述 2 条 `project_context.md` 中的核心偏好（不要只说"有偏好"）
- 复述 `alignment.md` 的关键字段（目标用户、核心方案、当前阶段范围）
- 检查 `迭代记录.md` 最近一段是否有「X→Y」「转向」「废弃」字样，有则在共识段加一句反向确认（例：「当前是年轻自媒体人，**不是**老人」）；无则跳过反向确认

第 4 点必须是可直接执行的子任务，不能是「继续推进 X」这种含糊话。

如果信息已经足够，不要再追问泛问题。

### 输出前自检清单
输出 4 段前，对照下面 5 条 yes/no 自检；任一条 no 就回到对应锚点重读：
1. 第 1 段是否一句话说出了"项目+当前阶段+在做什么"？
2. 第 2 段是否同时列出已完成与未完成？
3. 第 3 段是否包含 ≥2 条具体偏好 + alignment 关键字段？
4. 如近期有转向，第 3 段是否做了反向确认？
5. 第 4 段是否给出了具体可执行的子任务（动词开头）？

## 恢复规则

- 优先以 `alignment.md` 为当前共识
- 优先以 `project_context.md` 为背景和偏好
- 优先以 `next.md` 为当前动作
- 优先以 `迭代记录.md` 为最近变化
- 优先以 `task.md` 为已完成/未完成状态
- `实施计划.md` 默认不是必读文件
- 如果恢复后仍答不出上面的任一问题，明确指出缺的是哪一类信息
- 只有在锚点文件互相冲突时，才指出冲突并请用户拍板

## 锚点缺失降级

不是所有项目锚点文件都齐全。按下面降级，不要因为缺一个就拒绝恢复：

- `alignment.md` 缺：从 `迭代记录.md` 最近一段反推「当前共识看起来是 X，建议补 alignment 确认」
- `project_context.md` 缺：第 3 段只复述 alignment，明示「项目长期偏好尚未沉淀」
- `next.md` 缺：从 `task.md` 第一条进行中项推断下一步，并提示用户「next 缺失，请确认推断对不对」
- `task.md` 缺：第 2 段只能从迭代记录拼凑，明示信息不完整
- `迭代记录.md` 缺：无法反向确认转向，第 3 段跳过反向确认部分

降级时永远在输出末尾列出"缺失的锚点 + 建议补哪些"。

## 过滤过期信息

迭代记录里有大量"路径"信息（曾经的决策、曾经的方案、曾经的偏好）。这些是**轨迹**而不是**现状**，恢复时严防混进当前共识。

- `alignment.md` 是当前共识的**唯一权威**。迭代记录中任何与 alignment 冲突的内容，按"已被替换"对待，不要在输出第 3 点里复述
- 迭代记录中显式标注「X→Y」「转向」「废弃」「归档」字样的旧侧，全部不进现状
- 如果用户问"我们之前是怎么决定的"才补充历史路径；默认恢复只讲现状
- 任何不在 `alignment.md` / `project_context.md` / `next.md` / `task.md` 当前条目中的信息，都不进第 3 点

## 典型触发语

- `恢复会话`
- `继续上次`
- `先同步一下状态`
- `我们现在进行到哪了`
