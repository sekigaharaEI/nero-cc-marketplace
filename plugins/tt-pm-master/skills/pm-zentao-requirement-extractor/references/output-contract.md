# 输出合同

本文件约束筛选、命名和按功能分组输出。只有需要二次筛选、重命名、按功能分组，或发现输出样式漂移时读取。

## 默认产物

正常一次导出最多包含两类文件：

1. 原始字段提取文件：脚本直接生成，保留全量命中范围。
2. 筛选/按功能分组文件：当用户指定计划、指派人等筛选条件，或要求“梳理/分析”但没有明确要求报告时生成。

不要生成 `_分析_YYYY-MM-DD.md`、`需求分析`、`统计报告` 等分析文件。

如果误生成了分析报告或元数据文件，必须删除后再收尾。尤其不要留下：

- `*_分析_*.md`
- `*_分析报告_*.md`
- `*需求分析*.md`
- `export_metadata.json`

## 命名规则

优先使用稳定、口径清晰的命名：

- 原始字段提取：`禅道_<产品名>_研发需求_字段提取_<YYYY-MM-DD>.md`
- 计划筛选字段提取：`禅道_<产品名>_<计划名>_字段提取_<YYYY-MM-DD>.md`
- 计划筛选且按功能分组：`禅道_<产品名>_<计划名>_按功能分组_<YYYY-MM-DD>.md`

避免以下命名漂移：

- 不要用目录名或业务线简称替代禅道产品名，例如不要把产品 `业务系统线上反馈` 写成 `销售`。
- 不要在文件名里加入真实姓名，如 `指给涂磊`。筛选口径写在文件头即可。
- 不要在文件名里加入 `指给我`。
- 不要把筛选文件命名成 `研发需求_<计划名>_指给某某_<日期>.md`。
- 不要在用户未明确要求分析报告时生成或命名 `_分析_YYYY-MM-DD.md`。

## 字段提取文件头模板

字段提取或计划筛选文件头保持以下结构：

```markdown
# 禅道字段提取

- 来源页面：`<url 或 product=id>`
- 范围：产品 `<product_id>`「<product_name>」研发需求，共 `<total>` 条，已按 assignedTo=`<account_or_name>` 过滤，命中 `<assigned_count>` 条 → 再按计划=`<plan_name>` 过滤后保留 `<filtered_count>` 条
- 提取字段：研发需求名称、需求描述（含内嵌截图）、所属模块、当前状态、优先级、计划、创建日期
- 说明：所属模块优先取研发需求详情；若研发需求未挂模块，则补用源 Bug 模块；仍无法判断时标记为“未设置”
- 说明：截图按禅道原 HTML 位置内嵌在需求描述中，不再单列
- 说明：优先级取禅道 `pri` 字段，1 最高 / 2 高 / 3 中 / 4 低；计划优先取 `planTitle`，多个用顿号拼接，仅有计划 ID 时记为「未命名计划（#id）」
- 二次过滤：仅保留 `**计划**：<plan_name>` 的研发需求
```

如果没有二次过滤，删掉最后一行，并把范围行写成脚本默认范围。

不要改成：

```markdown
# 禅道字段提取 - <计划> 指给<姓名>
- 筛选口径：...
- 原始导出：...
- 命中条数：...
- 导出日期：...
```

## 按功能分组文件头模板

按功能分组文件使用字段提取文件头，并追加排序说明：

```markdown
- 排序：按"功能"语义聚合（同一功能的需求归到同一节），组之间按组内最高优先级降序、同档按规模降序排；组内按优先级 1(最高) → 4(低) 排序
```

随后输出：

```markdown
## <功能组名>

### <id>

**研发需求名称**：...
```

条目正文必须从字段提取文件复制，不改写、不摘要、不把截图另列。

## 用户说“分析”时的解释

在本 skill 中，以下表达默认解释为“导出可分析材料，必要时按功能分组”：

- “拉一份下来分析”
- “导出后分析”
- “梳理一下”
- “整理一下需求”

即使用户说了“分析”，本 skill 仍只产出字段提取和按功能分组文件。周报、总结、统计结论、风险建议应作为导出完成后的普通写作任务另行处理。
